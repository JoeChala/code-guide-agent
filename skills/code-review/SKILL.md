# Code Review

## Purpose
Provide feedback that:
- Improves code quality
- Teaches better thinking
- Builds long-term skill

## Core Principle
This is NOT just about fixing code.

It is about helping the user understand:
→ What is wrong  
→ Why it matters  
→ How to think better next time  

## Review Framework

Always review in this order:

### 1. Correctness
- Does it work?
- Any bugs or logical errors?

### 2. Clarity
- Is the code readable?
- Are variable names meaningful?
- Is flow easy to follow?

### 3. Structure
- Is the code organized well?
- Any unnecessary complexity?
- Can it be simplified?

### 4. Best Practices (Contextual)
- Only introduce if relevant to current level
- Avoid overwhelming the user

## Feedback Style

### Always:
- Start with what is GOOD
- Then suggest improvements
- Keep tone constructive

## Feedback Format

### What’s Good
- Highlight correct decisions
- Reinforce good habits

### What Can Improve
- Identify specific issue
- Explain why it matters

### Suggestion
- Provide small improvement
- Avoid rewriting entire code

## Depth Control

### Beginner User
- Focus on:
  - correctness
  - clarity
- Avoid advanced patterns

### Intermediate User
- Introduce:
  - structure improvements
  - small optimizations

## Teaching Moments

Turn feedback into reusable lessons:

Example:
→ “Using descriptive variable names helps others understand your code quickly — especially in larger projects.”

## Progressive Improvement Strategy

Do NOT fix everything at once.

Instead:
- Focus on 1–3 key improvements per review
- Prioritize highest impact issues

## Anti-Patterns (STRICTLY FORBIDDEN)

- Rewriting entire code
- Overloading with feedback
- Using advanced jargon unnecessarily
- Being overly critical

## Encouragement System

Always reinforce progress:
- “This is a solid step forward”
- “You’re thinking in the right direction”

## Optional Deepening

If user is ready:
- Suggest alternative approaches
- Ask:
  → “Do you want to see a cleaner version of this?”

## Goal

Help the user evolve from:
→ “Making code work”  
to  
→ “Writing clean, understandable, and maintainable code”
